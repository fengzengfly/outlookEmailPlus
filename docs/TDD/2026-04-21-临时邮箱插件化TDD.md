## 1. 文档目标

本文档仅讨论「临时邮箱 Provider 插件化」的**测试设计**，不重复展开业务需求与实现细节。

本 TDD 重点回答以下问题：

1. 新增的插件系统涉及哪些测试层（注册表 / 工厂 / 管理服务 / API / 前端）。
2. 如何在**不依赖真实第三方插件**的前提下，高可信验证插件加载、安装、配置、热刷新的完整链路。
3. 如何保证插件化改造不误伤现有 CF Worker / GPTMail 的内置 provider 链路（向后兼容）。
4. 如何验证安全隔离（插件加载失败不影响系统、卸载时活跃邮箱检查）。

## 2. 测试目标（必须同时成立）

1. **注册表正确**：装饰器能将内置 provider 注册到 `_REGISTRY`；工厂能从注册表正确查找和实例化。
2. **插件加载安全**：第三方 .py 文件能被动态加载；语法错误或 import 异常不影响内置 provider 和系统运行。
3. **热刷新有效**：刷新后第三方插件从注册表更新；内置 provider 白名单不受影响。
4. **生命周期完整**：安装 → 配置 → 卸载全链路可通过 API 验证，且错误码语义稳定。
5. **安全隔离**：卸载时正确阻止有进行中任务邮箱的插件；配置 key 按 `plugin.{name}.{field}` 隔离。
6. **向后兼容**：现有 temp-mail 全部 API（用户侧 + external 侧）路径不变、返回结构不变。

## 3. 测试原则

### 3.1 测试优先级（按风险排序）

1. **注册表 + 工厂核心逻辑**（装饰器注册、查表实例化、白名单保护）
2. **插件管理服务**（安装/卸载/配置读写/活跃邮箱检查）
3. **热刷新机制**（清空第三方、重新扫描、保护内置）
4. **Controller / API 接口**（路由注册、参数校验、错误码传播）
5. **CLI 命令**（install/uninstall/list 参数解析与调用）
6. **前端契约**（低风险，最小覆盖）

### 3.2 测试策略

遵循"**单元为主，接口集成为核心，CLI/前端冒烟兜底**"的策略：

- 单元测试：65%（注册表、工厂、管理服务、热刷新）
- 接口/集成测试：30%（API 端到端、与现有 temp-mail 的兼容性）
- CLI / 前端：5%（参数解析、最小冒烟）

### 3.3 明确不做（本期不强制）

1. 真实第三方插件的端到端测试（用模拟插件替代）
2. 并发热刷新压测
3. 浏览器级 E2E 自动化（用 Flask test_client + 合约为主）

## 7. 契约（Contract）校验点

### 7.1 安装响应契约

`POST /api/plugins/install` 成功响应：

```
必须包含: success=True, code="OK", data.plugin_name, data.file_path
可选包含: data.dependencies[]
```

### 7.2 卸载响应契约

`POST /api/plugins/{name}/uninstall` 成功响应：

```
必须包含: success=True, data.plugin_name
可选包含: data.had_active_emails (bool), data.cleaned_keys (string[])
```

### 7.3 配置 Schema 契约

`GET /api/plugins/{name}/config/schema` 成功响应：

```
必须包含: data.plugin_name, data.config_schema
data.config_schema.fields[] 每项必须包含: key, label, type
可选字段: required, placeholder, default, description, options
```

### 7.4 热刷新响应契约

`POST /api/system/reload-plugins` 成功响应：

```
必须包含: success=True, data.loaded(string[]), data.failed(object[])
data.failed[].name: string
data.failed[].error: string
```

### 7.5 向后兼容契约

以下现有接口在插件化改造后必须保持不变：

| 接口 | 不变项 |
|------|--------|
| `GET /api/temp-emails/options` | 返回结构不变 |
| `POST /api/temp-emails/generate` | 参数和返回结构不变 |
| `GET /api/temp-emails/{email}/messages` | 返回结构不变 |
| `POST /api/external/temp-emails/apply` | 参数和返回结构不变 |
| `GET /api/settings` | temp_mail_* 设置项不变 |

## 12. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| `importlib` 行为在不同 Python 版本有差异 | 加载测试不稳定 | 固定 Python 3.11+（项目要求） |
| 测试间 DB 状态污染 | 测试结果不稳定 | 每个测试前清理 settings 和 temp_emails 表 |
| `tmp_path` 在 Windows 上路径分隔符 | 文件路径错误 | 使用 `pathlib.Path` 统一路径处理 |
| mock registry 与真实 registry 不同步 | 安装测试通过但真实场景失败 | 层 H 端到端测试用真实 registry.json |
| 热刷新后 `_REGISTRY` 状态残留 | 测试间相互影响 | 热刷新测试后清理 `_REGISTRY` 并重新导入内置 provider |
