## 1. 文档目标

本文档仅讨论「CF 临时邮箱接入邮箱池」的**测试设计**，不重复展开业务需求与实现细节。

本 TDD 重点回答以下问题：

1. 这次改造需要测哪些层（Repository / Service / Controller / 前端契约）。
2. 动态创建 + 自动删除的关键路径，如何在**不依赖真实 CF Worker** 的前提下高可信验证。
3. 如何保证改造不误伤现有 Outlook / IMAP 邮箱池链路（向后兼容）。
4. 哪些风险必须用自动化测试覆盖，哪些仅做人工冒烟即可。

---

## 2. 测试目标（必须同时成立）

本次测试目标不是“接口能通”，而是证明以下四件事同时成立：

1. **动态创建可用**：当调用方指定 `provider=cloudflare_temp_mail` 且池中无可用邮箱时，系统能动态创建并返回可领取的邮箱（含 claim_token 等关键字段）。
2. **删除策略正确**：`complete` 时仅在 `result in ('success','credential_invalid')` 触发删除远程邮箱；其它结果不删；且删除失败不影响本地状态流转。
3. **兼容性不破坏**：不传 provider 或 provider 非 CF 时，原邮箱池行为不变（包括 project_key 防重复、email_domain 过滤、lease 超时回收等）。
4. **对外 API 契约稳定**：外部接口 `/api/external/pool/*` 返回结构保持兼容；新增 provider 参数不影响旧调用方。

---

## 3. 测试原则

### 3.1 测试优先级（按风险排序）

1. **Pool Repository 层原子逻辑**（claim 动态创建、complete 触发删除）
2. **Service 层参数校验与错误映射**（provider 白名单、上游错误码传播）
3. **External Pool API 接口集成**（provider 参数透传、权限、审计）
4. **与 external 读信/验证码提取的兼容性**（确保 CF 邮箱能走既有 mailbox_resolver 逻辑）
5. **前端契约（只读显示）**（低风险，最小覆盖）

### 3.2 测试策略

遵循“**单元为主，接口集成为核心，人工冒烟兜底**”的策略：

- 单元测试：60% ~ 70%
- 接口/集成测试：25% ~ 35%
- 前端契约/人工冒烟：5% ~ 10%

### 3.3 明确不做（本期不强制）

1. 真实 CF Worker 的端到端联调自动化（避免外部依赖与波动）
2. 压测级性能测试（可在后续 P1/P2 做）
3. 浏览器级完整 E2E 自动化（本期以 Flask test_client + 合约为主）

## 4. 测试分层设计

> 说明：本项目测试入口统一使用 `tests/_import_app.py`，其会将 `DATABASE_PATH` 指向临时 DB 文件，并禁用调度器自启动。

**目标**：确保 CF Provider 适配层可被稳定 mock，并且其自身行为在已有测试覆盖下保持稳定。

**现状**：已有 `tests/test_temp_mail_provider_cf.py`，覆盖 create/delete/list 等行为。

**本期建议新增/补充（可选）**：

| 用例 | 目的 | 备注 |
|---|---|---|

## 6. Mock 策略（关键）

### 6.1 禁止真实网络

所有涉及 CF Worker 的调用必须 mock：

- `CloudflareTempMailProvider.create_mailbox`
- `CloudflareTempMailProvider.delete_mailbox`
- （若 e2e 涉及读信）`CloudflareTempMailProvider.list_messages`

### 6.2 推荐 mock 方式

- `unittest.mock.patch` 直接 patch provider 方法
- 让 create_mailbox 返回固定结构：

```python
{
  "success": True,
  "email": "abc123@cf-domain.com",
  "meta": {"provider_jwt": "jwt", "provider_mailbox_id": "123"}
}
```

并针对失败场景返回：

```python
{"success": False, "error": "not configured", "error_code": "TEMP_MAIL_PROVIDER_NOT_CONFIGURED"}
```

## 7. 契约（Contract）校验点

### 7.1 claim 响应最小字段（当前路由: /api/external/pool/claim-random）

`/api/external/pool/claim`（概念名）/ `/api/external/pool/claim-random`（当前路由）成功响应 data 至少包含：

- `account_id`
- `email`
- `claim_token`
- `lease_expires_at`

新增能力：当请求 body 带 `provider=cloudflare_temp_mail` 时，返回仍满足以上字段（并可额外包含 provider 字段，视实现约定）。

### 7.2 向后兼容

不传 provider 时：

- 响应结构不变
- 行为不变
