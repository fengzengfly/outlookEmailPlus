## 1. 功能定义

V1 目标：通过路径统一重构 + 6 项性能优化，将 Outlook 验证码提取端到端耗时从 8.5-15.5s 降至 ≤4s。

### 1.1 路径统一重构（前置）

将 Web 端与外部 API 的 Outlook 验证码提取路径合并为一套统一的 service 层实现。

### 1.2 性能优化（6 项）

1. **Graph API 权限预检** — 跳过无权限的 Graph 调用
2. **IMAP 连接复用** — fetch+detail 共用一次 IMAP 会话
3. **IMAP OAuth Token 短期缓存** — 避免重复刷新 access_token
4. **通道缓存** — 记忆账号可用渠道，跳过已知不可用的
5. **IMAP 批量 FETCH** — 多封邮件一次网络往返
6. **IMAP 并发双服务器** — 竞速两台 IMAP 服务器

## 3. 系统行为设计

### 3.1 路径统一 — 验证码提取统一流程

#### 3.1.1 统一入口函数

新增 service 层统一函数（位于 `services/verification_channel_routing.py` 或独立文件）：

```
extract_verification_for_outlook(
    account: Dict,
    proxy_url: str,
    resolved_policy: Dict,
    code_source: str,
    expected_field: str | None = None,
    from_contains: str = "",
    subject_contains: str = "",
    since_minutes: int | None = None,
    baseline_timestamp: int | None = None,
) -> Dict[str, Any]
```

**返回值**：
```python
{
    "success": bool,
    "data": { ... },           # 提取结果（验证码/链接/formatted）
    "error": { ... },          # 错误信息（当 success=False）
    "channel_used": str,       # 命中的渠道
    "new_refresh_token": str,  # 如有轮换
}
```

#### 3.1.2 统一流程

```
1. build_verification_channel_plan(preferred_channel)
   → 生成有序 channel_plan: [preferred, ...其余]

2. for channel in channel_plan:
   2a. 通道缓存命中检查 → 已知不可用则 skip
   2b. Graph 权限预检 → 无 Mail.Read 则 skip graph_* 渠道
   2c. fetch_emails_for_channel(channel) → success?
       - 失败 → 记录 upstream_error, continue
       - 成功但空 → continue
   2d. (外部 API 专有) filter_messages(from/subject/since)
   2e. fetch_email_detail_for_channel(channel, message_id)
       ※ 当 channel 为 IMAP 时，使用连接复用版本
   2f. extract_verification_info_with_options()
   2g. enhance_verification_with_ai_fallback()
   2h. apply_confidence_gate(enforce_mutual_exclusion=False)
   2i. 判断是否命中 → 命中则更新 preferred_channel, 写通道缓存, return

3. 全部渠道遍历完毕 → 判断 auth_expired / email_not_found
```

> **⚠️ 行为变更说明（路径统一后）**：
> - **apply_confidence_gate**：Web 端当前未调用此函数，统一后将调用 → Web 端低置信度结果可能被过滤
> - **enforce_mutual_exclusion**：Web 端当前使用默认值 `True`（code 和 link 互斥），统一后使用 `False`（code 和 link 可共存）→ Web 端验证码提取可能同时返回 code 和 link
> - 以上行为对齐 External API 的现有行为，对注册场景更合理（同时获取验证码和验证链接）

#### 3.1.3 Web 端调用方式

`controllers/emails.py` `api_extract_verification`:

```python
# account_type == "imap" → 保持现有 imap_generic 路径
# account_type == "outlook" → 调用统一函数
result = extract_verification_for_outlook(
    account=account,
    proxy_url=proxy_url,
    resolved_policy=verification_policy,
    code_source=code_source,
)
# controller 只负责 HTTP 响应封装 + account_summary 更新
```

**预期代码变化**：controller 中 Outlook 路径从 ~600 行缩减至 ~50 行。

#### 3.1.4 外部 API 调用方式

`services/external_api.py` `_extract_verification_with_memory_for_outlook`:

```python
result = extract_verification_for_outlook(
    account=account,
    proxy_url=proxy_url,
    resolved_policy=resolved_policy,
    code_source=code_source,
    expected_field=expected_field,
    from_contains=from_contains,
    subject_contains=subject_contains,
    since_minutes=since_minutes,
    baseline_timestamp=baseline_timestamp,
)
```

**预期代码变化**：外部 API 的 ~130 行循环逻辑迁入统一函数，原函数变为薄包装。

### 3.2 Graph API 权限预检

#### 3.2.1 行为

1. `get_access_token_graph_result()` 返回值新增 `scope` 字段（从 token response 解析）
2. 新增工具函数 `has_mail_read_permission(scope: str) -> bool`
3. `get_emails_graph()` 在获取 token 后检查 scope：
   - 无 `Mail.Read` / `Mail.ReadWrite` → 直接返回 `{"success": False, "auth_expired": True, "no_mail_permission": True}`
   - 不发起 Graph messages API 请求
4. 通道缓存可选择记住 "该账号无 Graph 权限"

#### 3.2.2 scope 解析规则

MS token endpoint 返回的 `scope` 是空格分隔的字符串，例如：
```
"User.Read profile openid email"
```
（无 `Mail.Read` = 无邮件读取权限）

检测条件：`"Mail.Read" in scope or "Mail.ReadWrite" in scope`

### 3.3 IMAP 连接复用（fetch + detail 共用会话）

#### 3.3.1 行为

新增组合函数（`services/imap.py`）：

```
fetch_and_detail_imap_with_server(
    account: str,
    client_id: str,
    refresh_token: str,
    folder: str = "inbox",
    top: int = 1,
    server: str = IMAP_SERVER_NEW,
) -> Dict[str, Any]
```

**返回值**：
```python
{
    "success": bool,
    "emails": [...],         # 邮件摘要列表
    "detail": { ... },       # 最新一封的完整详情（body/body_html/raw_content）
    "error": { ... },        # 错误信息
}
```

#### 3.3.2 内部流程

```
1. get_access_token_imap_result() → access_token (※ 走缓存)
2. IMAP4_SSL(server) → authenticate(XOAUTH2)
3. SELECT folder
4. SEARCH ALL → message_ids
5. 取最新 top 封的 msg_id
6. FETCH msg_id (RFC822) → 解析为 headers + body_preview → emails[]
7. FETCH 最新 1 封的完整 RFC822 → 解析为 detail（含 body/body_html/raw_content）
8. logout()
```

步骤 6-7 在同一个 IMAP 连接内完成。如果 top=1，步骤 6 和 7 实际上是同一次 FETCH（只需解析出摘要和详情两种视图）。

#### 3.3.3 在 routing 层的集成

`verification_channel_routing.py` 新增：

```
fetch_emails_and_detail_for_channel(
    account, channel, proxy_url, skip, top
) -> Dict
```

当 channel 为 IMAP 时调用 `fetch_and_detail_imap_with_server()`，一次调用同时返回 emails + detail。

当 channel 为 Graph 时，仍分两步（Graph API 的 list 和 detail 是不同的 endpoint，共享 access_token 但无连接复用收益）。

### 3.4 IMAP OAuth Token 短期缓存

#### 3.4.1 行为

在 `services/imap.py` 模块级维护 token 缓存：

```python
_token_cache: Dict[str, Tuple[str, float]] = {}  # key → (access_token, expires_at)
_token_cache_lock = threading.Lock()
```

- **key**: `f"{client_id}:{hashlib.sha256(refresh_token.encode()).hexdigest()[:16]}"`
- **TTL**: `expires_in - 60` 秒（MS token 通常 `expires_in=3599`，即缓存 ~59 分钟）
- **命中**: 直接返回缓存的 access_token
- **未命中/过期**: 调用 MS token endpoint，成功后写入缓存

#### 3.4.2 失效场景

- refresh_token 被轮换 → key 自动变化（hash 不同），旧缓存自然过期
- 手动清除：`clear_imap_token_cache(client_id=None)` 清单个或全部

### 3.5 通道缓存（Channel Cache）

#### 3.5.1 行为

新建 `services/channel_capability_cache.py`：

```python
_cache: Dict[str, Dict[str, Any]] = {}  # email → { "available": ["imap_new", ...], "unavailable": ["graph_inbox", ...], "expires_at": float }
_cache_lock = threading.Lock()
```

- **写入时机**: `fetch_emails_for_channel()` 返回后，记录该渠道成功/失败
- **读取时机**: channel_plan 遍历前，过滤掉已知不可用的渠道
- **TTL**: 1 小时
- **失效触发**: token 刷新成功 / 账号凭据变更

#### 3.5.2 与 preferred_channel 的职责分工

| 维度 | preferred_channel | 通道缓存 |
|------|-------------------|----------|
| 存储 | DB（持久化） | 进程内存（易失） |
| 语义 | "上次成功的渠道" → 优先尝试 | "已知可用/不可用" → 跳过不可用 |
| 更新时机 | 验证码提取成功时 | 每次渠道调用后 |
| 作用 | 排序 channel_plan | 过滤 channel_plan |

### 3.6 IMAP 批量 FETCH

#### 3.6.1 行为

修改 `get_emails_imap_with_server()` 的 FETCH 逻辑：

**现有**：
```python
for msg_id in paged_ids:
    status, msg_data = connection.fetch(msg_id, "(RFC822)")
```

**优化后**：
```python
ids_str = b",".join(paged_ids)
status, all_data = connection.fetch(ids_str, "(RFC822)")
```

一次网络往返获取所有邮件。

#### 3.6.2 兼容性

- 单封（top=1）: `connection.fetch(b"42", "(RFC822)")` → 行为不变
- 多封: `connection.fetch(b"40,41,42", "(RFC822)")` → 一次返回多条
- 返回数据结构: `all_data` 是交错的 `[(header, body), b")", (header, body), b")", ...]`，需按对解析

### 3.7 IMAP 并发双服务器

#### 3.7.1 行为

新增 `get_emails_imap_concurrent()` 函数：

```python
def get_emails_imap_concurrent(
    account, client_id, refresh_token, folder, skip, top,
    servers=("outlook.live.com", "outlook.office365.com"),
) -> Dict[str, Any]:
```

使用 `concurrent.futures.ThreadPoolExecutor(max_workers=2)`：
1. 两个线程分别连接两台服务器
2. `as_completed()` 取第一个成功结果
3. 另一个连接 `logout()` 关闭

#### 3.7.2 与通道缓存的交互

- 通道缓存命中（已知 imap_new 可用）→ 只连单服务器，不并发
- 通道缓存未命中 → 并发两台
- 结果写回通道缓存

## 7. 错误处理

### 7.1 统一函数的错误返回

| 场景 | 错误码 | HTTP 状态 |
|------|--------|-----------|
| 所有渠道都失败 + Graph auth_expired + IMAP 未连通 | `ACCOUNT_AUTH_EXPIRED` | 401 |
| 所有渠道成功但无邮件 | `EMAIL_NOT_FOUND` | 404 |
| 有邮件但提取不到验证信息 | `VERIFICATION_NOT_FOUND` | 404 |
| IMAP 连接成功但信箱为空 | `EMAIL_NOT_FOUND` | 404 |

### 7.2 降级策略

- Token 缓存过期 → 自动刷新，透明降级
- 通道缓存不可用 → 退化为完整 channel_plan 遍历（无缓存等价于当前行为）
- 并发双服务器两个都失败 → 返回最后一个错误
