## 当前真实状态（不要再重复分析）

以下结论已经在本会话中确认：

### 1. 后端协议已经足够

当前已存在并可复用的插件配置接口：

- `GET /api/plugins/{name}/config/schema`
- `GET /api/plugins/{name}/config`
- `POST /api/plugins/{name}/config`
- `POST /api/plugins/{name}/test-connection`

**结论**：本次任务**不需要新增插件配置后端协议**。

### 2. `/api/settings` 不适合承载插件 schema 配置

`static/js/main.js` 中的 `saveSettings()` 当前只处理：

- `temp_mail_provider`
- `temp_mail_*`
- `cf_worker_*`

**结论**：插件自己的 `config_schema` 配置应继续通过 `/api/plugins/{name}/config` 保存，**不要把插件配置硬塞进 `/api/settings`**。

### 3. 当前前端的两个真实缺口

#### 缺口 A：临时邮箱页域名选择仍写死在 `cloudflare_temp_mail`

文件：`static/js/features/temp_emails.js`

- `loadTempEmails()`
- `onTempEmailProviderChange(selectedProvider)`

当前逻辑仍是：

```js
domainSelect.disabled = providerSelect.value !== 'cloudflare_temp_mail';
```

所以第三方插件即使返回 `domains`，页面上也不能手动选域名。

#### 缺口 B：插件配置仍内嵌在“插件管理”卡片里

文件：`static/js/features/plugins.js`

- `toggleConfig(name)`
- `_renderConfigForm(...)`

当前仍把插件配置渲染到 `plugin-cfg-{name}` 这类卡片内联区域中，导致：

- 插件管理 = 安装 / 卸载 / 应用变更
- 插件配置 = base_url / token / domains / timeout

两类职责混在一起。

### 4. 设置页已有天然承载位

文件：`templates/index.html`

当前设置页临时邮箱 Tab 中已经有：

- `#gptmailConfigPanel`
- `#cfWorkerConfigPanel`
- `#pluginManagerCard`

**推荐落点**：在 `#cfWorkerConfigPanel` 与 `#pluginManagerCard` 之间新增：

- `#pluginProviderConfigPanel`

### 5. 设置页 Provider 切换逻辑也已明确改造点

文件：`static/js/main.js`

函数：`onTempMailProviderChange(provider)`

当前行为：

- `legacy_bridge` → 显示 GPTMail 面板
- `cloudflare_temp_mail` → 显示 CF Worker 面板
- 插件 provider → 两张内置面板都隐藏

**结论**：本次方案 B 的设置页入口，应在这个函数中接入插件设置面板的显示逻辑。

## 必须遵守的实现边界

1. **不要新增插件配置后端协议**
   - 复用现有 `/api/plugins/{name}/config*` 接口
2. **不要把插件 schema 配置并入 `/api/settings`**
   - `/api/settings` 只继续负责：
     - `temp_mail_provider`
     - `temp_mail_*`
     - `cf_worker_*`
3. **不要改产品路径**
   - `/api/temp-emails/*`
   - `/api/external/temp-emails/*`
4. **不要破坏内置配置页**
   - GPTMail 面板保留
   - CF Worker 面板保留
5. **不要破坏插件管理卡片现有生命周期能力**
   - 安装
   - 卸载
   - 应用变更
   - `load_failed` 错误展示
6. **不要只修域名下拉**
   - 这是方案 A，已被否决
   - 你必须按方案 B 同时处理：
     - 设置页插件配置承载区
     - 插件管理职责收口
     - 临时邮箱页域名泛化

### Phase 3：临时邮箱页域名选择泛化（后做）

#### 任务 3.1：去掉 `cloudflare_temp_mail` 硬编码

修改 `static/js/features/temp_emails.js`：

当前硬编码点：

- `loadTempEmails()`

改造目标：

1. 根据当前 provider 的 `get_options()` 返回决定：
   - 是否有 `domains`
   - 是否允许手动选域名
   - 提示文案是什么
2. 不再直接按 provider 名称判断

---

#### 任务 3.2：引入按 Provider 维度的 options 状态管理

当前 `tempEmailOptionsCache` 是单槽缓存。

请改为至少满足以下任一策略：

1. `Map<provider_name, options>`
2. 或显式记录：
   - 当前缓存属于哪个 provider
   - 缓存何时失效

**目标**：

- 切换不同 provider 时不会复用错误的旧 options
- 域名下拉 / hint / error 状态与当前 provider 一致

---

#### 任务 3.3：统一域名下拉与提示文案

你需要统一处理：

- `#tempEmailDomainSelect`
- `#tempEmailOptionsHint`
- `#tempEmailOptionsStatus`

要求至少覆盖 3 类状态：

1. **支持手动域名**
   - 域名下拉启用
   - options.domains 渲染可选项
2. **不支持手动域名**
   - 下拉禁用
   - 显示 provider 对应提示，不要一律写成 GPTMail 文案
3. **options 拉取失败**
   - 显示错误提示
   - 不误导成“该 provider 天生不支持域名”
