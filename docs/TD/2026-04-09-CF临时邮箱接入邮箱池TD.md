## 2. 当前技术现状

### 2.1 邮箱池现状

**核心模块**:
- `outlook_web/services/pool.py`: 业务逻辑层,负责输入校验、配置读取、错误转换
- `outlook_web/repositories/pool.py`: 数据访问层,负责原子事务操作
- `outlook_web/controllers/external_pool.py`: 外部 API 路由层

**数据模型**(Schema v19):
```sql
-- accounts 表关键字段
provider TEXT DEFAULT 'outlook'  -- 'outlook', 'imap', 'custom'
pool_status TEXT                 -- 'available', 'claimed', 'used', 'cooldown', ...
claim_token TEXT
claimed_by TEXT
claimed_at TEXT
lease_expires_at TEXT
```

**已支持功能**:
- ✅ 随机领取 available 邮箱
- ✅ Project 级防重复领取
- ✅ Email domain 过滤
- ✅ 租约超时自动回收
- ✅ 完成后状态流转(success → used, timeout → cooldown)

**不支持的场景**(本期新增):
- ❌ 池中无可用邮箱时,动态创建新邮箱
- ❌ CF 临时邮箱的 JWT 等元数据存储
- ❌ 使用完成后自动删除远程 CF 邮箱

### 2.2 CF Provider 现状

**核心模块**:
- `outlook_web/services/temp_mail_provider_cf.py`: CF Worker API 适配器
- `outlook_web/services/mailbox_resolver.py`: 邮箱类型识别与 mailbox 对象构建

**已支持功能**:
- ✅ 创建 CF 邮箱: `create_mailbox(prefix, domain)` → `{email, meta: {provider_jwt, ...}}`
- ✅ 删除 CF 邮箱: `delete_mailbox(mailbox)` → `True/False`
- ✅ 读取邮件: `list_messages(mailbox)` → `[{message_id, subject, content, ...}]`
- ✅ 元数据存储在 `meta` 对象中(provider_jwt, provider_mailbox_id, address_id)

**当前存储位置**: `temp_emails` 表(与 accounts 表隔离)

## 3. 核心技术决策

### 3.1 决策一: 统一表模型 vs 双表适配器

**选定方案**: 统一表模型(方案A)

**实施方式**:
- CF 邮箱写入 `accounts` 表,设置 `provider='cloudflare_temp_mail'`
- 新增 `temp_mail_meta` 列存储 CF 特有元数据(JSON 格式)
- `temp_emails` 表保持只读,历史数据不迁移

**理由**:
1. 邮箱池现有逻辑 100% 依赖 accounts 表,双表适配器会大幅增加复杂度
2. CF 邮箱需要 pool_status / claim_token 等池管理字段,temp_emails 表无这些列
3. 统一表模型使 CF 邮箱与 Outlook/IMAP 邮箱享受完全相同的池调度能力
4. `temp_mail_meta` JSON 列扩展性强,未来可支持更多 provider

**示例数据**:
```json
{
  "id": 123,
  "email": "abc123@cf-domain.com",
  "provider": "cloudflare_temp_mail",
  "pool_status": "claimed",
  "temp_mail_meta": "{\"provider_jwt\":\"eyJ...\",\"provider_mailbox_id\":\"456\"}",
  "claim_token": "clm_xxx",
  "claimed_by": "reg_script_001:task_001",
  "claimed_at": "2026-04-09T10:00:00Z"
}
```

### 3.2 决策二: 动态创建 vs 预生成池

**选定方案**: 动态创建模式

**实施方式（以当前代码为准）**:
- `Service.claim_random()` 先调用 `Repository.claim_atomic()` 尝试从池中领取
- 若池空且 `provider='cloudflare_temp_mail'`：
  - Service 层调用 `_create_cf_mailbox_for_pool()`（内部调用 `CloudflareTempMailProvider.create_mailbox`）
  - 再调用 `Repository.insert_claimed_account()` 写入 `accounts`（`pool_status='claimed'`）
- 无需管理员预先批量创建

**理由**:
1. CF Worker 创建邮箱速度快(1-2s),动态创建用户体验可接受
2. 无需占用 CF Worker 配额预生成大量闲置邮箱
3. 用完即删,资源利用率高
4. 代码实现简单,无需调度器/容量管理

### 3.3 决策三: 删除策略

**选定方案**: 智能删除策略

**删除时机**(在 `complete_claim()` 中判断):
- ✅ `result='success'` → 删除远程 CF 邮箱(注册成功,邮箱无用)
- ✅ `result='credential_invalid'` → 删除远程 CF 邮箱(账号异常,清理资源)
- ❌ `result='verification_timeout'` → **不删除**(可能需要人工介入查看邮件)
- ❌ `result='network_error'` → **不删除**(临时错误,保留现场)
- ❌ `result='provider_blocked'` → **不删除**(平台风控,保留现场)

**非阻塞设计**:
- 删除失败仅记录 WARNING 日志,不影响本地 pool_status 更新
- 本地状态已流转到 `used` / `retired`,业务流程正常完成

**失败兜底**(P2,本期不做):
- 定期扫描 `provider='cloudflare_temp_mail' AND pool_status='used'` 且创建时间超过 7 天的记录
- 调用 CF Provider 删除远程邮箱
- 记录删除结果到审计日志

## 5. Pool 核心逻辑实现

### 5.1 Repository 层扩展

**文件**: `outlook_web/repositories/pool.py`

**新增函数（以当前实现为准）**: `insert_claimed_account()`

**函数签名（摘录）**:
```python
def insert_claimed_account(
    conn: sqlite3.Connection,
    *,
    email: str,
    caller_id: str,
    task_id: str,
    lease_seconds: int,
    provider: str,
    account_type: str = "temp_mail",
    project_key: Optional[str] = None,
    temp_mail_meta: Optional[dict] = None,
    claim_log_detail: str = "动态创建",
) -> dict:
    """插入一个新账号并直接标记为 claimed（Repository 层仅 DB 写入，函数内部包含 BEGIN IMMEDIATE/COMMIT）。"""
```

**核心逻辑（Repository 仅 DB）**:
1. `BEGIN IMMEDIATE`
2. `INSERT accounts`（写入 `pool_status='claimed'`，并写入 `temp_mail_meta` 明文 JSON）
3. `INSERT account_claim_logs`（action='claim'，detail 可由上层传入如“CF邮箱动态创建”）
4. 可选：写入/更新 `account_project_usage`
5. `COMMIT`

**关键技术细节**:

**Q1**: `client_id` 和 `refresh_token` 字段如何填充?
- **A**: CF 邮箱无 OAuth 凭据,这两个字段填空字符串 `''`(数据库 NOT NULL 约束)

**Q2**: `email_domain` 字段如何填充?
- **A**: 从 `result['email']` 中提取 `@` 后的部分,小写存储

**Q3**: CF Worker 创建失败如何处理？
- **A**: 上游网络调用在 Service 层，创建失败会转换为 `PoolServiceError`（error_code 如 `UPSTREAM_TIMEOUT/UPSTREAM_SERVER_ERROR`）。Repository 层不直接处理上游网络错误。

**示例返回值**:
```python
{
    "id": 123,
    "email": "abc123@cf-domain.com",
    "provider": "cloudflare_temp_mail",
    "pool_status": "claimed",
    "temp_mail_meta": '{"provider_jwt":"eyJ...","provider_mailbox_id":"456"}',
    "claim_token": "clm_abc123xyz",
    "claimed_at": "2026-04-09T10:00:00Z",
    "lease_expires_at": "2026-04-09T10:10:00Z",
    "email_domain": "cf-domain.com"
}
```

### 5.2 claim_atomic() 扩展

**文件**: `outlook_web/repositories/pool.py`

**现有逻辑**(伪代码):
```python
def claim_atomic(conn, caller_id, task_id, lease_seconds, provider=None, ...):
    sql = "SELECT * FROM accounts WHERE pool_status='available' AND status='active'"
    if provider:
        sql += " AND provider = ?"
    sql += " ORDER BY RANDOM() LIMIT 1"
    
    conn.execute("BEGIN IMMEDIATE")
    account = conn.execute(sql, params).fetchone()
    
    if account is None:

        return None  # 无可用邮箱
    
    # 更新 account 为 claimed 状态
    # 返回完整 account 对象
```

**扩展逻辑（以当前实现为准）**:
```python
def claim_atomic(conn, caller_id, task_id, lease_seconds, provider=None, ...):
    sql = "SELECT * FROM accounts WHERE pool_status='available' AND status='active'"
    if provider:
        sql += " AND provider = ?"
    sql += " ORDER BY RANDOM() LIMIT 1"
    
    conn.execute("BEGIN IMMEDIATE")
    account = conn.execute(sql, params).fetchone()
    
    if account is not None:
        # 有可用邮箱,直接 claim(现有逻辑不变)
        # ... 更新 account 为 claimed 状态 ...
        conn.execute("COMMIT")
        return account
    
    # 无可用邮箱：Repository 返回 None（不在 Repository 层动态创建）
    
    # 其他 provider 无可用 → 回滚并返回 None
    conn.execute("ROLLBACK")
    return None
```

**关键技术细节**:

**Q1**: 池空时为什么不在 Repository 层动态创建？
- **A**: 分层约束：Repository 只做 SQLite 原子操作；动态创建属于上游网络调用，放在 Service 层实现，避免跨层依赖。

**Q2**: 动态创建耗时 1-2s,是否会阻塞其他 claim 请求?
- **A**: 会。因为使用 `BEGIN IMMEDIATE` 事务,整个 claim 操作持有写锁。但这是合理的 — 池调度本身就是串行化的,避免竞态条件(如两个请求同时 claim 同一账号)。CF Worker 创建耗时可接受(1-2s),不会产生明显性能问题。

**扩展逻辑**(伪代码):
```python
def complete(conn, account_id, claim_token, caller_id, task_id, result, detail):
    # 1. 获取 account 信息(需要 provider 和 temp_mail_meta)
    account = conn.execute(
        "SELECT * FROM accounts WHERE id = ?", (account_id,)
    ).fetchone()
    
    # 2. 更新 pool_status(现有逻辑不变)
    new_pool_status = RESULT_TO_POOL_STATUS[result]
    conn.execute("BEGIN IMMEDIATE")
    conn.execute("UPDATE accounts SET pool_status = ?, ...", ...)
    conn.execute("INSERT INTO account_claim_logs ...", ...)
    conn.execute("COMMIT")
    
    # 3. 如果是 CF 邮箱 + 需要删除 → 调用 CF Provider 删除远程邮箱
    if account['provider'] == 'cloudflare_temp_mail':
        if _should_delete_cf_mailbox(result):
            _delete_cf_remote_mailbox_nonblocking(account)
    
    return new_pool_status
```

**新增辅助函数**:

```python
def _should_delete_cf_mailbox(result: str) -> bool:
    """判断是否需要删除远程 CF 邮箱"""
    return result in ('success', 'credential_invalid')

def _delete_cf_remote_mailbox_nonblocking(account: dict) -> None:
    """
    非阻塞删除远程 CF 邮箱(仅记录日志,不抛异常)
    
    Args:
        account: accounts 表记录(含 email, temp_mail_meta)
    """
    import logging
    logger = logging.getLogger(__name__)
    
    try:
        # 解析 temp_mail_meta
        meta_raw = account.get('temp_mail_meta') or '{}'
        meta = json.loads(meta_raw)
        
        # 构建 mailbox 对象
        mailbox = {'email': account['email'], 'meta': meta}
        
        # 调用 CF Provider 删除
        from outlook_web.services.temp_mail_provider_cf import CloudflareTempMailProvider
        provider = CloudflareTempMailProvider()
        success = provider.delete_mailbox(mailbox)
        
        if success:
            logger.info(
                f"[pool] 已删除 CF 远程邮箱: {account['email']} "
                f"(account_id={account['id']})"
            )
        else:
            logger.warning(
                f"[pool] 删除 CF 远程邮箱失败(返回 False): {account['email']} "
                f"(account_id={account['id']})"
            )
    except Exception as e:
        logger.warning(
            f"[pool] 删除 CF 远程邮箱异常: {account['email']} "
            f"(account_id={account['id']}), error={e}"
        )
        # 非阻塞,继续执行,本地状态已更新
```

**关键技术细节**:

**Q1**: 为什么删除操作在事务外执行?
- **A**: 避免删除失败导致整个 complete 事务回滚 — 本地状态更新(pool_status 流转)是核心业务,必须成功;远程邮箱删除是"尽力而为",失败可接受(会留下垃圾资源,但不影响业务流程)。

**Q2**: 如果 CF Worker 删除接口返回 500,是否重试?
- **A**: 本期不重试,仅记录 WARNING 日志。未来可通过定期清理脚本兜底(P2)。

def claim_random(..., provider=None):
    _validate_caller_id(caller_id)
    _validate_task_id(task_id)
    project_key = _validate_project_key(project_key)
    email_domain = _validate_email_domain(email_domain)

                f"provider 必须是 {sorted(VALID_PROVIDERS)} 之一",

            )
    
    conn = create_sqlite_connection()
    try:
        settings = _read_settings_via_conn(conn)
        default_lease = settings["pool_default_lease_seconds"]
        _validate_lease_seconds(default_lease)
        
        account = pool_repo.claim_atomic(
            conn,
            caller_id=caller_id,
            task_id=task_id,
            lease_seconds=default_lease,
            provider=provider,  # ← 传递 provider 参数
            project_key=project_key,
            email_domain=email_domain,
        )
        
        if account is None and provider == 'cloudflare_temp_mail':
            # 池空 + CF：动态创建（Service 层网络调用）→ Repository 仅 DB 写入
            created_email, created_meta = _create_cf_mailbox_for_pool(email_domain=email_domain)
            account = pool_repo.insert_claimed_account(
                conn,
                email=created_email,
                caller_id=caller_id,
                task_id=task_id,
                lease_seconds=default_lease,
                provider='cloudflare_temp_mail',
                account_type='temp_mail',
                project_key=project_key,
                temp_mail_meta=created_meta,
                claim_log_detail='CF邮箱动态创建',
            )
            return account
        if account is None:
            raise PoolServiceError("池中没有符合条件的可用邮箱", "no_available_account", http_status=200)
        return account
    except pool_repo.PoolRepositoryError as e:
        # Repository 层抛出的异常(如 CF Worker 创建失败)
        # 转换为业务错误码返回给客户端
        raise PoolServiceError(str(e), e.error_code, http_status=500)
    finally:
        conn.close()

**新增错误码**:
- `TEMP_MAIL_PROVIDER_NOT_CONFIGURED`: CF Worker 未配置(base_url 或 admin_key 缺失)
- `UPSTREAM_TIMEOUT`: CF Worker 请求超时(30s)
- `UPSTREAM_SERVER_ERROR`: CF Worker 返回 5xx
- `invalid_provider`: 传入的 provider 值不合法

### 7.2 /api/external/pool/claim-complete 扩展

**实现对齐说明（以当前实现为准）**：

- Repository 层的 `complete()` 仅负责本地状态流转与 claim log 写入
- Service 层的 `complete_claim()` 在事务提交后，根据 `provider == 'cloudflare_temp_mail'` 且 `result in ('success','credential_invalid')` 执行非阻塞远程删除

**行为变化**(对客户端透明):
- 如果 `account.provider == 'cloudflare_temp_mail'` 且 `result in ('success', 'credential_invalid')`,后台会自动删除远程 CF 邮箱
- 删除失败不影响 API 响应（仍返回 200 + `{"success": true, "code": "OK", "message": "success", "data": {"account_id": 123, "pool_status": "used"}}`，字段以实际 controller 实现为准）

## 10. 错误处理与日志

### 10.1 错误码定义

| 错误码 | HTTP状态 | 说明 | 触发场景 |
|--------|---------|------|----------|
| `invalid_provider` | 400 | provider 参数值不合法 | 传入非白名单的 provider 值 |

| `no_available_account` | 200 | 池中无可用邮箱(含动态创建失败) | claim 时池空且 CF Worker 创建失败 |

## 11. 性能与安全

### 11.1 性能指标

| 指标 | 目标值 | 实测方法 |
|------|--------|----------|
| claim 响应时间(无可用邮箱) | < 3s | 压测工具(100 QPS) |
| claim 响应时间(有可用邮箱) | < 100ms | 压测工具(100 QPS) |
| complete 响应时间 | < 500ms | 压测工具(100 QPS) |
| 并发创建 CF 邮箱 | 支持 10 QPS | 压测工具(10 并发线程) |

**性能优化点**:
- CF Worker 创建邮箱耗时 1-2s,不做并行优化(单线程创建足够)
- 删除操作在 `complete()` 事务外执行,不阻塞响应

### 11.2 安全措施

**输入校验**:
- `provider` 参数白名单校验(`VALID_PROVIDERS`)
- `caller_id` / `task_id` 长度限制(64/128 字符)
- `email_domain` 格式校验(域名规范)

**权限控制**:
- `/api/external/pool/claim-random` 需要 `X-API-Key` 鉴权
- `/api/external/pool/claim-random` 需要 `pool_access` 权限(在 `external_api_keys` 表中配置)

**资源隔离**:
- CF 邮箱与 Outlook/IMAP 邮箱在同一 accounts 表,通过 `provider` 字段区分
- `project_key` 防止同项目重复领取(已有机制,无需新增)

**审计日志**:
- 所有 claim / complete 操作记录到 `account_claim_logs` 表
- CF 邮箱删除结果记录到应用日志(文件 `server.log`)

## 13. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| CF Worker 配额耗尽 | 无法动态创建新邮箱 | 返回明确错误码,提示管理员检查 CF Worker 配额 |
| CF Worker 服务不可用 | claim 失败 | 返回 `UPSTREAM_SERVER_ERROR`,调用方可重试或切换其他 provider |
| 删除远程邮箱失败 | 垃圾资源残留 | 非阻塞设计,记录 WARNING 日志,定期人工清理(P2) |
| Schema 迁移失败 | 应用无法启动 | 迁移前打印备份提示,失败时抛出异常并记录 trace_id |
| 账号列表性能下降 | 页面加载慢 | CF 邮箱与 Outlook/IMAP 共用 `idx_accounts_status` 索引,查询性能不变 |
