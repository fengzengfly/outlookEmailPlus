# BUG-00013 对外开放 API 公网暴露风险与可用性缺口

> 关联需求: `PRD-00008` / `FD-00008` / `TDD-00008`
> 分析时间: 2026-03-11
> 分析范围: `outlook_web/services/external_api.py`、`outlook_web/controllers/emails.py`、`outlook_web/controllers/system.py`、`outlook_web/security/auth.py`、`Dockerfile`

---

## 一、问题概述

| 编号 | 标题 | 严重程度 | 类型 |
|------|------|---------|------|
| **E-1** | 对外开放 API 已具备功能，但尚不适合直接公网暴露 | 🔴 高 | 架构 / 可用性 / 安全 |

---

## 二、结论摘要

当前项目的 `/api/external/*` 已经完成了接口实现、`X-API-Key` 鉴权、设置页接入、OpenAPI 草稿和专项测试，**可以用于单一可信调用方、受控私有部署下的试运行**。

但如果目标是“作为真实对外 API 长期稳定暴露给外部系统接入”，当前实现仍存在明显缺口：

1. **可用性层面**：同步长轮询接口与单 worker 部署模型叠加，容易阻塞整个服务。
2. **安全层面**：只有 API Key，没有限流、白名单、配额或调用方隔离。
3. **运维层面**：健康检查与账号状态检查偏静态，不能证明真实上游可读。
4. **产品层面**：当前是“单实例单 Key 读全部邮箱”模型，只适合自用，不适合作为多方接入平台。

因此，本问题不是“接口是否存在”，而是**接口是否已经达到可对外暴露的工程标准**。当前答案是：**功能上可用，工程上未达公网开放标准**。

### 3.2 未满足项

PRD-00008 的真实目标并不只是“把接口做出来”，而是满足：

- 本地化部署后的自动化注册流程稳定接入
- 自动提取失败后的人工排查
- 本地部署完成后的连通性与可用性确认

当前与这个目标的偏差主要体现在：

| 目标 | 当前情况 | 风险 |
|------|----------|------|
| 对外长期稳定提供服务 | 单 worker + 同步轮询 | 容易阻塞、吞吐低 |
| 可控对外暴露 | 仅 API Key | Key 泄露后攻击面过大 |
| 自检能证明真实可用 | 多为静态检查 | 可能“自检成功但实际不可读” |
| 面向外部系统接入 | 单 Key 读全实例邮箱 | 权限边界过粗 |

## 四、根因分析

### 4.1 单 worker 与 `wait-message` 同步轮询叠加

当前 Docker 默认使用 Gunicorn 单 worker：

```dockerfile
CMD ["gunicorn", "-w", "1", "-b", "0.0.0.0:5000", "--timeout", "120", "--access-logfile", "-", "web_outlook_app:app"]
```

同时，`wait-message` 首版采用同步轮询实现，最大可阻塞 120 秒：

```python
while True:
    latest_message = get_latest_message_for_external(...)
    if int(latest_message.get("timestamp") or 0) >= baseline_timestamp:
        return latest_message

    if time.time() - start >= timeout_seconds:
        raise MailNotFoundError(...)

    time.sleep(poll_interval)
```

这会导致：

1. 一个长轮询请求就可能占满唯一 worker。
2. Graph / IMAP 上游变慢时，普通查询接口也会排队。
3. 后台页面与外部 API 共用同一个进程池，互相影响。

这类实现可用于“内网低频自用”，但不适合真实公网暴露。

### 4.2 鉴权只有 API Key，没有第二道防线

当前开放接口仅做：

- Header 中读取 `X-API-Key`
- 与全局 `external_api_key` 比较

缺少：

- IP 白名单
- 外部接口限流
- 单调用方配额
- 多 API Key 隔离
- 按邮箱范围授权
- 公网模式下的接口分级开关

因此当前模型本质上是：

> 一个 Key = 全实例全部已配置邮箱的只读能力

这适合“自己调用自己的本地服务”，不适合“提供给多个外部系统接入”。

### 4.3 自检接口偏静态，不证明真实上游可读

`/api/external/health` 目前主要检查：

- 服务是否存活
- SQLite 是否可访问

`/api/external/account-status` 主要检查：

- 账号是否存在
- 状态是否为 disabled
- 凭证字段是否为空

但它没有真正做：

- Graph 连通性探测
- IMAP 登录探测
- 最近一封邮件读取探测

因此会出现：

| 场景 | 当前结果 | 实际情况 |
|------|----------|----------|
| refresh token 已失效 | `account-status` 可能显示 `can_read=true` | 实际读取失败 |
| IMAP 密码字段存在但错误 | `account-status` 可能显示可读 | 实际读取失败 |
| 上游 Graph 网络故障 | `health` 仍返回服务正常 | 实际外部调用不可用 |

### 4.4 文档与发布定位还停留在草案/内部交付阶段

OpenAPI 文档当前仍为 `0.1.0-draft`，并明确标注为草稿。

README 当前也没有完整覆盖：

- 外部 API 接入方式
- 反向代理/TLS 示例
- 生产模式安全建议
- `wait-message` 的并发影响
- 建议禁用/限制的公网接口

这说明当前能力更像“项目内可交付功能”，还不是“可对外发布产品”。

## 六、可借鉴方案分析

### 6.1 `exeample/outlookEmail`：可借鉴“最小外部 API 闭环”，但不能照搬安全方案

该示例已经实现了一个最小可用外部接口：

- `GET /api/external/emails`
- 设置页支持 `external_api_key`
- 有单独的 `docs/api.md`

优点：

| 项目 | 可借鉴点 |
|------|----------|
| 接口范围控制 | 首版只开放一个外部查询接口，范围很收敛 |
| 文档闭环 | 有明确 curl 示例和错误响应说明 |
| 设置页接入 | 用户能直观看到如何配置和使用 |

不足：

| 项目 | 不可照搬点 |
|------|------------|
| API Key 传递 | 同时支持 Header 和 query 参数 `api_key`，会增加泄露风险 |
| 存储方式 | `external_api_key` 为明文读取 |
| 能力边界 | 只有最小接口，未处理公网暴露时的工程问题 |

结论：

- **可借鉴**：外部 API 需要独立文档、最小首发能力、清晰调用示例。
- **不可照搬**：query 传 key、明文存储、过于简化的安全模型。

### 6.2 `exeample/BillionMail`：可借鉴“外围防护优先”思路

`BillionMail` 虽然不是同类读取 API 项目，但它在对外暴露上有两个非常实用的外围防护思路：

1. `FAIL2BAN_INIT=y`
2. `IP_WHITELIST_ENABLE=false/true`

这说明它把“能否被访问”放在应用逻辑之外的一层去控制：

| 机制 | 含义 | 对我们可借鉴之处 |
|------|------|------------------|
| Fail2Ban | 对异常访问做封禁 | 外部接口应有暴力访问抑制 |
| IP 白名单开关 | 限制管理入口来源 IP | 对外 API 应支持白名单模式 |

结论：

- 我们当前项目缺少的，不只是代码层鉴权，还缺**外围防护开关**。
- 对于自部署场景，**反向代理白名单 + 应用层 API Key** 的双层模型明显更稳。

### 6.3 `exeample/MailAggregator_Pro`：可借鉴“后台轮询与 Web 接口解耦”

`MailAggregator_Pro` 的核心特点不是开放 API，而是把轮询任务放到后台 worker/poller 中：

- `main.py` 在启动时创建 `poller_loop()`
- `worker/poller.py` 持续异步轮询
- `api/health.py` 暴露轮询状态

这类设计对我们有两个启发：

| 方向 | 可借鉴点 |
|------|----------|
| 长耗时任务 | 不应该依赖请求线程同步等待 |
| 健康检查 | 应暴露“后台轮询是否正常工作”的状态，而不只是进程存活 |

结论：

- `wait-message` 如果要保留长期能力，应逐步改成“后台轮询 / 事件缓存 / 异步等待”模式。
- 对外健康检查应补充“最近轮询时间、最近一次错误、上游探测结果”等动态指标。

## 七、建议解决方案

### 7.1 总体策略

不建议把当前版本直接定义为“公网开放 API”，建议分三阶段推进：

| 阶段 | 定位 | 目标 |
|------|------|------|
| P0 | 受控私有接入 | 仅供单一可信调用方使用 |
| P1 | 半开放部署 | 具备外围防护与接口分级 |
| P2 | 正式外部能力 | 具备多调用方与更强可用性能力 |

在具体实现上，建议始终保留两条主线：

1. **鉴权收敛**：继续坚持 `X-API-Key` 单入口，后续再逐步补公网模式、来源白名单、限流、多 Key 与范围授权。
2. **轮询解耦**：`wait-message` 不应长期依赖请求线程同步等待，后续应逐步迁移到后台轮询、事件缓存、异步等待或独立 worker 模式。

## 八、分阶段实施方案

### 8.1 P0：先收敛发布口径，明确“仅适合受控私有接入”

建议立即补充文档与部署口径：

1. README 明确声明：
   - 当前外部 API 适用于本地化部署、单实例、单可信调用方。
   - 不建议直接暴露在公网。
2. OpenAPI/FD/README 中标记：
   - `wait-message`
   - `/raw`
   为高风险/高成本接口。
3. 补充反向代理建议：
   - 必须使用 HTTPS
   - 建议仅内网访问或白名单访问

这是最快能落地、风险最低的动作。

### 8.2 P1：补外围防护与接口分级

建议新增以下能力：

| 能力 | 说明 | 参考来源 |
|------|------|----------|
| `EXTERNAL_API_PUBLIC_MODE` | 公网模式开关 | 当前项目新增 |
| `EXTERNAL_API_IP_WHITELIST` | 支持来源 IP 白名单 | 借鉴 `BillionMail` |
| 外部接口限流 | 按 IP / API Key 限制请求频率 | 借鉴 `BillionMail` 的 fail2ban 思路 |
| 接口分级控制 | 公网模式禁用 `/wait-message`、`/raw` | 当前项目新增 |

建议规则：

```text
私有模式:
  允许全部 external API

公网模式:
  默认禁用 /api/external/wait-message
  默认禁用 /api/external/messages/{id}/raw
  必须启用白名单或限流
```

### 8.3 P1：增强自检接口，让“可读性”变成真实探测

建议新增或改造：

| 接口 | 当前 | 建议 |
|------|------|------|
| `/api/external/health` | 服务/DB 存活 | 增加上游探测摘要 |
| `/api/external/account-status` | 静态字段判断 | 增加真实读取探测结果 |
| `/api/external/capabilities` | 能力列表 | 增加 `public_mode`、`restricted_features` |

建议返回增加：

- `upstream_probe_ok`
- `probe_method`
- `last_probe_at`
- `last_probe_error`
- `public_mode`
- `restricted_features`

### 8.4 P2：重构 `wait-message`，避免阻塞请求线程

建议不要长期保持“请求线程里 while + sleep”的模式，后续应改为以下之一：

1. 后台轮询缓存
2. 事件队列 / 消息中转
3. 独立 worker + 轮询结果表

这一点可以参考 `MailAggregator_Pro` 的思路：

- Web API 负责查询状态
- Worker 负责持续轮询与更新结果

这一方向应视为后续“**轮询解耦**”主线，而不是可有可无的优化项。

如果暂时不做完整异步化，最低也应：

- 在公网模式禁用 `wait-message`
- 私有模式下明确其高成本属性

### 8.5 P2：补权限模型，避免“一把 Key 读全实例”

建议后续设计：

| 能力 | 作用 |
|------|------|
| 多 API Key | 区分不同调用方 |
| Key 级邮箱白名单 | 限制某个 Key 只能读部分邮箱 |
| Key 状态管理 | 支持启停、轮换、吊销 |
| 调用审计增强 | 记录调用方标识、命中邮箱范围 |

当前单 `external_api_key` 模式适合首版，但不适合作为长期外部平台方案。

这一方向应视为后续“**鉴权收敛**”主线。

## 十、最终判断

### 10.1 当前是否“可用”

**可用。**

适用于：

- 本地部署
- 单一可信调用方
- 内网或受控访问环境
- 自动化脚本低频调用

### 10.2 当前是否“可对外暴露”

**不建议直接公网暴露。**

原因不是功能缺失，而是工程能力还未闭环：

1. 长轮询会占用单 worker。
2. 缺少限流、白名单、配额等防护层。
3. 自检接口不能证明真实可读。
4. 单 Key 读全实例的权限边界过粗。

### 10.3 推荐定位

建议当前版本的正式定位为：

> “面向本地化部署与单可信调用方的受控开放 API”

而不是：

> “可直接公网暴露的通用开放平台 API”
