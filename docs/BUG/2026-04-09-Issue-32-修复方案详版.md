# Issue #32 修复方案详版（代码级）

> 对应问题：
> 1) 分组页“全选”仅作用于当前分页
> 2) 删除账号在存在池化历史时返回 `ACCOUNT_DELETE_FAILED` / 500

---

## 0. 复审结论（基于实际代码二次审查）

本方案主方向正确。二次审查后，结合你已确认的策略（**全选=整个分组全部账号**），结论如下：

1. **全选作用域固定为“当前分组全量账号”**（已确认）
   - 作用域应基于 `accountsCache[currentGroupId]`，而不是当前页 DOM。
   - 这与 Issue #32 诉求完全一致。
2. **删除路径优先级应以 `/api/accounts/<id>` 为主**
   - 当前前端实际删除入口（`deleteAccount(accountId, email)`）只调用 `/api/accounts/${accountId}`。
   - `/api/accounts/email/<email>` 路由存在，但当前 UI 未使用，应作为兼容路径测试而非主路径。
3. **搜索模式需要显式策略**
   - `searchAccounts(query)` 当前渲染的是“全局搜索结果”，并非单分组视图。
   - 既然全选策略已定为“全分组”，则搜索模式下建议隐藏或禁用“全选”开关，避免用户误以为是在全选搜索结果。

以下章节已按上述结论修订。

## 3.3 具体改动点

### A. 新增“当前分组作用域”辅助函数

在 `groups.js` 增加（命名可微调）：

```javascript
function getCurrentAccountScopeIds() {
    const source = Array.isArray(accountsCache[currentGroupId])
        ? applyFiltersAndSort(accountsCache[currentGroupId])
        : [];
    return source
        .map(acc => Number(acc && acc.id))
        .filter(id => Number.isInteger(id) && id > 0);
}

function getCurrentAccountScopeStats() {
    const scopeIds = getCurrentAccountScopeIds();
    const selectedInScope = scopeIds.filter(id => selectedAccountIds.has(id)).length;
    return {
        scopeCount: scopeIds.length,
        selectedInScope,
        allSelected: scopeIds.length > 0 && selectedInScope === scopeIds.length,
        noneSelected: selectedInScope === 0,
    };
}
```

> 说明：
> - 作用域固定来自“当前分组全量账号”（`accountsCache[currentGroupId]`），严格贴合 issue。
> - 当 `accountsCache[currentGroupId]` 为空时，可降级使用当前 DOM 复选框，防止首次加载前按钮误操作。
> - 搜索模式建议禁用全选，防止作用域认知错位。

### B. 改造 `selectAllAccounts()` / `unselectAllAccounts()`

由“遍历当前页 checkbox”改为“遍历当前作用域 ids”：

```javascript
function selectAllAccounts() {
    const scopeIds = getCurrentAccountScopeIds();
    scopeIds.forEach(id => selectedAccountIds.add(id));
    syncActiveCheckboxesFromSelection();
    updateBatchActionBar();
    updateSelectAllCheckbox();
}

function unselectAllAccounts() {
    const scopeIds = getCurrentAccountScopeIds();
    scopeIds.forEach(id => selectedAccountIds.delete(id));
    syncActiveCheckboxesFromSelection();
    updateBatchActionBar();
    updateSelectAllCheckbox();
}
```

### C. 新增 `syncActiveCheckboxesFromSelection()`

用于把当前页渲染 checkbox 与 `selectedAccountIds` 对齐：

```javascript
function syncActiveCheckboxesFromSelection() {
    const checkboxes = getActiveAccountCheckboxes();
    checkboxes.forEach(cb => {
        const id = Number(cb.value);
        cb.checked = selectedAccountIds.has(id);
    });
}
```

### D. 改造 `updateSelectAllCheckbox()`

由“当前 DOM checked 计数”改为“当前作用域已选计数”：

```javascript
function updateSelectAllCheckbox() {
    const stats = getCurrentAccountScopeStats();
    const selectAllCheckboxes = [
        document.getElementById('selectAllCheckbox'),
        document.getElementById('compactSelectAllCheckbox')
    ].filter(Boolean);

    selectAllCheckboxes.forEach(selectAllCheckbox => {
        if (stats.scopeCount === 0 || stats.noneSelected) {
            selectAllCheckbox.checked = false;
            selectAllCheckbox.indeterminate = false;
            return;
        }
        if (stats.allSelected) {
            selectAllCheckbox.checked = true;
            selectAllCheckbox.indeterminate = false;
            return;
        }
        selectAllCheckbox.checked = false;
        selectAllCheckbox.indeterminate = true;
    });
}
```

### E. 渲染后同步

在 `renderAccountList()` / `renderCompactAccountList()` 末尾继续调用：

- `syncActiveCheckboxesFromSelection()`（可选，若 HTML 已按 `selectedAccountIds` 输出可省略）
- `updateSelectAllCheckbox()`
- `updateBatchActionBar()`

并在空状态分支中，避免手工设置 `selectAllCheckbox.indeterminate = selectedAccountIds.size > 0`，
改为统一调用 `updateSelectAllCheckbox()`，确保状态以“当前分组作用域”计算。

### G. 搜索模式处理（新增）

建议在 `groups.js` 增加 `isSearchMode` 状态：

- `searchAccounts(query)` 成功后：`isSearchMode = true`
- 清空搜索并恢复分组列表时：`isSearchMode = false`
- `toggleSelectAll()` 开头加守卫：


这样可保证策略一致：**全选只针对当前分组，不针对搜索结果。**

## 4.3 具体改动点

### A. repository 引入 logger

```python
import logging
logger = logging.getLogger(__name__)
```

### B. `delete_account_by_id()` 改为事务删除

建议实现：

```python
def delete_account_by_id(account_id: int) -> bool:
    db = get_db()
    try:
        db.execute("DELETE FROM account_claim_logs WHERE account_id = ?", (account_id,))
        db.execute("DELETE FROM account_project_usage WHERE account_id = ?", (account_id,))
        cursor = db.execute("DELETE FROM accounts WHERE id = ?", (account_id,))
        db.commit()
        return cursor.rowcount > 0
    except Exception:
        try:
            db.rollback()
        except Exception:
            pass
        logger.exception("delete_account_by_id failed: account_id=%s", account_id)
        return False
```

### C. `delete_account_by_email()` 复用 by_id

```python
def delete_account_by_email(email_addr: str) -> bool:
    db = get_db()
    try:
        row = db.execute("SELECT id FROM accounts WHERE email = ?", (email_addr,)).fetchone()
        if not row:
            return False
        return delete_account_by_id(int(row["id"]))
    except Exception:
        logger.exception("delete_account_by_email failed: email=%s", email_addr)
        return False
```

> 好处：单点维护删除链路，避免两处逻辑漂移。

### D. controller 层保持 API 协议稳定

- `api_delete_account` / `api_delete_account_by_email` / `api_batch_delete_accounts` 返回结构不变。
- 异常细节走日志，不暴露给前端。
