> 目标：当 external_pool 调用 claim-random 且 `provider=cloudflare_temp_mail` 时：

> - 若池空：动态调用 CF Provider 创建，并写入 accounts（直接 claimed 状态）

**文件**：`outlook_web/services/pool.py`

- [x] 未知 provider：返回 `PoolServiceError(..., error_code='invalid_provider')`

**检查点**：
- [x] 不传 provider 时行为不变
- [x] provider 为空串时等价于 None

### Task 2.2：实现 `_claim_cf_dynamic_create()`（Service 私有函数）

**文件**：`outlook_web/services/pool.py`

输入：caller_id/task_id/default_lease/project_key/email_domain

步骤建议：

3. 组装写入 accounts 的字段（关键点见 Task 2.3）

> 目标：accounts 中的 CF pool 邮箱能被 external `/messages/latest`、`/verification-code` 读取。

### Task 4.1：修改 `mailbox_resolver.resolve_mailbox()`

**文件**：`outlook_web/services/mailbox_resolver.py`

- [ ] 当 accounts 命中且 `account['provider'] == 'cloudflare_temp_mail'`：
  - 返回 `kind='temp'` 的 mailbox 描述（而不是 kind='account'）
  - meta 从 `accounts.temp_mail_meta` 解析
  - 强制 `meta['provider_name'] = 'cloudflare_temp_mail'`（若缺失）
  - 设置 `mailbox_type='user'`（不绑定 consumer_key）
  - `status='active'`

**检查点**：
- [ ] `external_api.ensure_external_email_access()` 对该邮箱不再走 `ensure_account_can_read()`
- [ ] external list_messages_for_external() 能走 TempMailService

### Task 4.2：处理 accounts 与 temp_emails 同邮箱冲突

现有逻辑：accounts 与 temp_emails 同邮箱会抛 `MailboxConflictError`。

- [ ] 对 CF pool 邮箱：原则上不应写入 temp_emails 表，因此不应冲突
- [ ] 若线上存在冲突，暂不特殊处理（保持显式冲突更安全）

### Task 4.3：Claim token baseline 行为

- [ ] 确认 `pool_repo.get_claim_context()` 能返回 CF pool 账号的 `claimed_at`
- [ ] external read 接口的 `claim_token` baseline 不需要额外改动

### Task 4.4：缓存落库位置

当前 temp mail 的消息缓存写入 `temp_email_messages`，且依赖 `temp_emails` 表存在邮箱记录。

- [ ] 需要确认：当 mailbox_resolver 返回 kind=temp 但 temp_emails 表无该 email 时，`TempMailService.list_messages()` 会报 `TEMP_EMAIL_NOT_FOUND`
- [ ] 解决策略（实现时二选一）：
  1) **推荐**：为 CF pool 邮箱在 `temp_emails` 表创建一条不可见记录（visible_in_ui=0, mailbox_type='user'），meta 从 temp_mail_meta 拷贝
  2) 或改造 TempMailService 支持“无 temp_emails 记录也可直接读并缓存”（改动更大）

> 说明：该点属于“前后联调”核心阻塞，需优先验证。
