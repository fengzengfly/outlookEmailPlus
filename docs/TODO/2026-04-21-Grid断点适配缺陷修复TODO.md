## 关键发现：生产代码实际状态

### 旧 Grid 布局系统已废弃
- `layout-manager.js`、`layout.css`、`state-manager.js` 已被标记为 **Deprecated**（`tests/layout-system/README.md`）
- `test_smoke_contract.py` 明确断言这些文件**不应**出现在 HTML 中
- BUG 文档中分析的 Grid 四栏布局（`layout.css:44-58`）**不是当前生产代码**

### 当前生产布局：`.workspace` flex 三栏
- 模板：`templates/index.html` 中使用 `<div class="workspace workspace-mailbox">`
- CSS：`static/css/main.css` 中 `.workspace { display: flex; }`
- 三栏结构：
  - `.groups-column { width: 220px; flex-shrink: 0; }`
  - `.accounts-column { width: 280px; flex-shrink: 0; }`
  - `.emails-column { flex: 1; min-width: 0; }`

### 实际断点现状

| 区间 | 侧边栏 | workspace | 问题 |
|------|--------|-----------|------|
| ≥1025px | 220px 完整 | flex row，三栏固定宽度 | ✅ 正常 |
| **769px~1024px** | **60px icon-only** | **flex row，groups 180px / accounts 240px** | **❌ 问题区间** |
| ≤768px | 抽屉隐藏 | flex column，width 100% | ✅ 正常 |

### 769px~1024px 区间的实际根因
- `@media (max-width: 1024px) and (min-width: 769px)` 只改了 sidebar（缩为 60px）
- `.groups-column { width: 180px; }` 和 `.accounts-column { width: 240px; }` 仍为固定宽度
- `.workspace` 保持 `flex-direction: row`，没有折叠/隐藏机制
- 三栏最小总宽度 = 180 + 240 + emails内容(约300px+) = **720px+**
- 可用宽度 = 1024 - 60(sidebar) = 964px，看似足够...
- 但当窗口缩小到 **800px~900px** 时：可用宽度 = 800 - 60 = 740px，三栏内容（特别是 emails-column 内的搜索栏、+按钮、分页）开始挤压错位

## 响应式设计目标（用户要求）

基于用户反馈"应该是响应式的或者是pwa考虑一下"，修复不应只是调整两个断点数值，而应建立**完整的响应式断点体系**：

### 响应式策略草案

**桌面端（≥1025px）**
- 三栏全部展开，用户可拖拽调整宽度
- sidebar 完整 220px

**平板横屏/小窗（769px~1024px）**
- sidebar 缩为 60px（已有）
- **workspace 应有自适应能力**：
  - 方案 A：允许 `.groups-column` 和 `.accounts-column` 收缩（改 `flex-shrink: 0` → 允许收缩）
  - 方案 B：在此区间自动折叠 groups 面板，只显示 accounts + emails
  - 方案 C：根据可用宽度动态决定展开/折叠（JS 介入）

**平板竖屏/大手机（481px~768px）**
- 当前已有 `@media (max-width: 768px)`：sidebar 抽屉、workspace column
- 但可能需要更细致的处理（如隐藏某些列、简化表头）

**手机（≤480px）**
- 当前移动端适配基本可用
- 可考虑进一步优化（如全屏邮件详情、底部操作栏）

### PWA 考虑（可选）
- 添加 Web App Manifest（`manifest.json`）
- 支持 `display: standalone` 模式下的视口高度（`100dvh` 替代 `100vh`）
- 考虑底部安全区域（env(safe-area-inset-bottom)）

**目标**：在 `main.css` 的平板断点中增加 `.workspace` 的响应式处理（groups 折叠 + 展开入口）。

**修改位置**：
- `static/css/main.css` 第 1337–1352 行（`@media (max-width: 1024px) and (min-width: 769px)`）

- `static/js/main.js` 增加 `toggleGroupsColumn()`
- `static/js/i18n.js` 增加「展开分组」词条

**候选方案**：

#### 方案 A：允许收缩（纯 CSS，改动最小）
```css
@media (max-width: 1024px) and (min-width: 769px) {
  /* ... 现有 sidebar 规则 ... */
  .groups-column { width: 180px; flex-shrink: 1; min-width: 140px; }
  .accounts-column { width: 240px; flex-shrink: 1; min-width: 180px; }
  .emails-column { min-width: 280px; }
}
```
- 优点：纯 CSS，无需 JS
- 缺点：三栏同时被压缩，内容仍可能溢出

#### 方案 B：平板断点折叠 groups（CSS + 小量 JS）✅ **已选定**
```css
@media (max-width: 1024px) and (min-width: 769px) {
  /* ... 现有 sidebar 规则 ... */
  .workspace.workspace-mailbox .groups-column { display: none; }
  .workspace-resizer[data-resize-role="groups-to-accounts"] { display: none; }
  .accounts-column { width: 220px; }
}
```
- 优点：直接减少一栏，空间充足
- 缺点：需要 JS 配合提供"展开 groups"的入口（已实现）

**用户决策记录**：2026-04-21，用户明确选择方案 B。
